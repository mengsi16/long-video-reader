# 2026-06-17 长视频上下文覆盖诊断计划

## 问题

| 用户观察 | 风险判断 |
|---|---|
| 课程时长约 3 小时，但模型回答只覆盖一小段时间 | 聊天上下文可能只传入了转录开头和前若干关键帧 |
| 模型回答提到“约 14-15 分钟”“20 分钟左右” | 输入数据可能让模型误判视频总覆盖范围 |
| 用户提出向量检索、关键帧命名、目录打包 | 需要判断当前全量传入策略是否已经到达上下文瓶颈 |

## 冒烟检查

| 检查项 | 方法 |
|---|---|
| 转录文本规模 | 读取 `data/transcripts/4.txt` 的字符数、开头、中段、结尾 |
| 关键帧规模 | 调用 `/api/videos/4` 查看关键帧数量和时间跨度 |
| 当前模型输入规模 | 调用 `build_chat_messages()` 统计文本长度和图片数量 |
| 裁剪策略 | 审查 `backend/minimax_client.py` 与 `backend/main.py` |

## 判断标准

| 结果 | 结论 |
|---|---|
| 转录长于当前裁剪上限 | 当前确实没有传够内容 |
| 关键帧只取前 N 张 | 后半程视觉信息没有进入模型 |
| 当前问题是全局总结 | 需要全局索引或分段摘要，不适合只送开头 |

## 方案候选

| 方案 | 适用场景 |
|---|---|
| 扩大转录裁剪与均匀抽帧 | 临时缓解，适合小改动 |
| 分段文本摘要缓存 | 适合长视频全局问答与总结 |
| 转录分块检索 | 适合用户问局部概念、时间点、关键词 |
| 关键帧注释索引 | 适合 PPT/屏幕录制类课程，减少重复传图片 |
| 文本检索加关键帧目录打包 | 适合当前 3 小时课程类视频 |

## 输出

| 产物 | 内容 |
|---|---|
| 诊断结论 | 是否没传够、具体丢了哪些范围 |
| 修复建议 | 最小修复与中期结构方案 |
| 如需实施 | 先改最小范围，再做端到端验证 |
